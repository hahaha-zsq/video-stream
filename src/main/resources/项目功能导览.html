Project Overview / Media Middleware

RTSP 实时转发 与 文件转码能力 集中到一个服务里

`Video Stream Middleware` 是一个基于 Spring Boot、 Netty 与 JavaCV / FFmpeg 的音视频处理中间件。它一方面解决浏览器无法直接播放 RTSP 的问题，将实时流转换成 HTTP-FLV；另一方面提供 MP3 抽取、GIF 截取、加密 HLS 打包 等文件级处理能力，适合放在业务系统后面，作为统一的媒体能力层。

这个项目不是完整的视频平台，也不是单纯的播放器页面，而是一层把“摄像头直播中转”“上传后转码处理”“临时文件管理”和“资源自动回收”统一起来的后端服务。

这个项目解决了什么问题

从浏览器不能播 RTSP，到业务系统需要统一的媒体能力层
浏览器不支持 RTSP

摄像头能推 RTSP，但 Web 页面不能直接播放，前端需要依赖额外网关或中转服务。

多人观看浪费资源

如果每个用户都去单独拉一路 RTSP，会重复占用摄像头连接和服务器转码资源。

文件处理逻辑分散

MP3 抽取、GIF 截取、切片打包常常散落在多个脚本里，不利于统一接入和维护。

转码线程难管理

实时流一旦无人观看，如果没有回收机制，RTSP 长连、缓冲区和编码器会持续占用资源。

前端接入门槛高

业务前端不应直接理解 FFmpeg、RTSP、HLS 密钥和切片细节，更适合通过 HTTP 服务接入。

Feature 01

RTSP 实时转 HTTP-FLV

这是项目最核心的能力。客户端通过 GET /live?deviceId=...&rtspUrl=... 发起请求，服务端从摄像头拉取 RTSP 流，实时编码成浏览器更友好的 HTTP-FLV 输出。

  • 输出内容类型为 video/x-flv。
  • 采用分块传输，适合连续直播流推送。
  • 前端可直接使用 flv.js 进行播放。
Feature 02

流复用与广播推送

同一设备被多人观看时，系统不会重复创建多个转码线程，而是按 deviceId 复用同一个 TransferToFlv 任务，并通过 MediaChannel 广播给所有订阅客户端。

  • 减少摄像头侧连接压力。
  • 减少服务器 CPU 与带宽消耗。
  • 新客户端加入时会先补发 FLV Header。
Feature 03

文件上传后异步转码

项目不仅支持直播流，还对外提供文件级处理接口。上传视频或音频文件后，服务端通过异步任务执行转码，不阻塞 Web 主线程。

  • /api/convert/async-to-mp3：抽取音轨并输出 MP3。
  • /api/video/gif：按时间区间截取 GIF。
  • /api/video/convert：生成加密 HLS ZIP 包。

技术亮点

基于 README 与实际代码整理
Encoding

H.264 + AAC

实时流输出采用浏览器兼容度较高的 H.264 视频编码与 AAC 音频编码。

Latency

低延迟转码

录制器使用 preset=ultrafast 与 tune=zerolatency 追求实时性。

Transport

RTSP over TCP

抓流时显式设置 rtsp_transport=tcp，提升网络不稳定场景下的可靠性。

IO Model

Netty 非阻塞 I/O

实时流接口由 Netty 承载，适合处理长连接和高并发分发场景。

Reuse

任务复用

转码任务通过 ConcurrentHashMap 管理，按设备维度复用，避免重复工作。

Lifecycle

自动回收

当所有客户端断开后自动关闭转码线程、RTSP 连接与客户端通道，防止资源泄漏。

系统分层

客户端层、中间件层、数据源层
  • 客户端层：浏览器、移动端或其他 HTTP 客户端，负责发起播放请求或文件上传请求。
  • 中间件层：Spring Boot 控制器、Netty Server、LiveHandler、MediaStreamService、TransferToFlv、MediaChannel。
  • 数据源层：RTSP 摄像头、上传的本地音视频文件、临时工作目录中的转码产物。

核心组件职责

按 README 的模块说明补充整理
  • NettyServer：启动 Boss / Worker 线程组，配置监听端口、缓冲区和写水位线。
  • LiveHandler：校验 /live 请求、解析 deviceId 与 rtspUrl、发送 FLV 响应头。
  • MediaStreamService：维护 deviceId -> TransferToFlv 的映射关系，统一管理任务生命周期。
  • TransferToFlv：真正执行 RTSP 抓流、FLV 编码和数据推送。
  • MediaChannel：维护客户端连接集合，向多个订阅者广播 FLV 数据。
  • VideoConvertService / AudioConvertService：负责 HLS、GIF、MP3 等文件转码任务。

实时流工作流程

从请求进入，到转码、广播、释放资源
Step 01

客户端请求直播流

客户端请求 /live?deviceId=xxx&rtspUrl=xxx，Netty 收到完整 HTTP 请求后交给 LiveHandler 处理。

Step 02

查询是否已有任务

LiveHandler 从 MediaStreamService 查询当前 deviceId 是否已有运行中的转码任务。

Step 03

创建或复用转码任务

如果任务已存在，则直接把当前客户端加入 MediaChannel；如果不存在，则新建 TransferToFlv 和 MediaChannel 并启动转码线程。

Step 04

RTSP 抓帧并编码为 FLV

TransferToFlv 建立 RTSP 连接，抓取视频帧和音频帧，使用 H.264/AAC 编码后写入字节缓冲区，再推送给客户端。

Step 05

客户端离开后自动回收

当 MediaChannel 检测到连接集合为空时，发布 CloseStreamEvent，由 MediaStreamService 停止任务并释放资源。

接口能力总表

结合代码中的 Controller 与 README 整理
接口 默认端口 输入 输出 说明
GET /live 4901 deviceId、rtspUrl HTTP-FLV 流 把摄像头 RTSP 地址实时转成浏览器可播放的 FLV 流。
POST /api/convert/async-to-mp3 4900 音频或视频文件 MP3 下载 提取音轨并输出 MP3，适合录音归档和后续转写。
POST /api/video/gif 4900 视频文件、开始秒数、结束秒数 GIF 下载 最大 10 秒，内部会控制分辨率和帧率，避免输出过大。
POST /api/video/convert 4900 视频文件 ZIP 下载 生成加密 HLS 切片包，ZIP 中主要包含 .m3u8 与 .ts。
GET /api/video/key 4900 taskId、authCode 二进制密钥 为加密 HLS 回放提供密钥读取接口。

运行与配置

以当前代码中的配置为准
  • JDK 版本：Java 21。
  • 主 HTTP 端口：server.port=4900。
  • Netty 端口：netty-video.server.port=4901。
  • 上传限制：multipart.max-file-size=-1，当前不限制上传大小。
  • 临时目录：temp_path 指定上传源文件、GIF、MP3、HLS 切片与密钥的落盘位置。
  • 环境变量：支持通过 SPRING_PROFILES_ACTIVE、TEMP_PATH、NETTY_PORT 等变量覆盖。
server.port=4900 netty.port=4901 profile=dev/prod

编码与抓流参数

README 与实际代码中的关键项
  • 输出格式：flv。
  • 视频编码：H.264。
  • 音频编码：AAC。
  • 低延迟参数：preset=ultrafast、tune=zerolatency。
  • 质量控制：crf=25。
  • 帧率与 GOP：frameRate=25、gopSize=25。
  • RTSP 传输：优先走 tcp，并设置超时、探测和缓冲参数。

适用场景

更适合作为中间件服务，而不是完整视频平台
  • 安防监控 Web 化接入，需要把 RTSP 摄像头流转成浏览器可播放格式。
  • 业务系统希望快速接入 MP3、GIF、HLS 等媒体处理能力，而不想自己维护 FFmpeg 脚本。
  • 内网系统需要一个轻量、单体式、可快速部署的媒体能力网关。
  • 演示环境、项目 PoC、交付型项目中，需要快速打通“摄像头直播 + 文件转码”两条链路。

当前实现中的注意点

这些细节在接入和上线前需要确认
  • VideoConvertService 中 HLS 密钥地址仍写死为 localhost:8080，而当前 Spring Boot 默认端口实际为 4900。
  • 密钥接口的 authCode 当前写死为 secret123，适合演示，不适合直接用于生产环境。
  • temp_path 默认是开发机路径，部署到服务器时必须改成服务器本地可写目录。
  • 上传大小未限制，生产环境应配合网关、磁盘容量与权限策略一起控制风险。
  • 当前页面是说明页，不会替代项目里的实际播放测试页；直播接入建议仍结合 src/main/resources/play.html 验证。
